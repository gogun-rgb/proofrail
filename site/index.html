From claim to Verdict

Keep the evidence chain legible.

Proofrail does not turn a confident sentence into authority. It keeps each record distinct, checks the declared boundary, and lets deterministic rules decide what follows.

Input

Claims stay proposed.

An agent can say a change is ready. That statement can motivate an evaluation, but it cannot satisfy an Evidence Requirement by itself.

Collection

Facts keep their scope.

Authorized inspection produces Observations. Bounded commands produce Verification Receipts. Each record names its target, method, and lineage.

Decision

Rules derive the outcome.

Policy evaluates the declared inputs and emits one canonical Verdict: ADMISSIBLE, REVISION_REQUIRED, REJECTED, or BLOCKED.

Four canonical outcomes

A result with a reason, not a score.

These values describe the exact evaluation captured in an Evidence Bundle. They are not confidence labels and do not imply a general release decision.

  • ADMISSIBLE

    Declared requirements are met.

    The authorized target, scope, Observations, and Receipts satisfy the applicable deterministic Rules.

  • REVISION_REQUIRED

    Something needs another pass.

    Evidence is insufficient or a declared requirement is not satisfied, while the evaluation remains within its boundary.

  • REJECTED

    A rule denies this change.

    The evaluated scope or policy condition fails in a way that cannot be treated as an admissible result.

  • BLOCKED

    The determination cannot continue.

    Authority, target identity, execution boundary, or system integrity is not sufficient for a trustworthy determination.

Install once, inspect every run

A workflow you can read before you trust it.

The public prototype uses a reusable workflow, a base-branch configuration, and explicit bounded commands. The snippet below is intentionally copyable and contains no credentials.

  1. 01Place the configuration at .proofrail/config.yml on the base branch.
  2. 02Call the pinned workflow from the repository that owns the pull request.
  3. 03Read the summary and retain the Evidence Bundle for the exact evaluated head.

The Proofrail job appears as GitHub's automatic workflow check. The prototype runs on the caller's GitHub-hosted runner and does not ask for API keys, paid services, or a secret-bearing target environment.

workflow.yml + .proofrail/config.yml select to copy 
name: Proofrail
on:
  pull_request:
    types: [opened, synchronize, reopened, ready_for_review]
permissions:
  contents: read
  pull-requests: read
  checks: read
  statuses: read
jobs:
  proofrail:
    name: Proofrail
    uses: gogun-rgb/proofrail/.github/workflows/proofrail.yml@166573bc5ea5c32e7f8f0ed0943e67e466df9fcb
    with:
      config-path: .proofrail/config.yml
      strict: true
    permissions:
      contents: read
      pull-requests: read
      checks: read
      statuses: read

# Base branch file: .proofrail/config.yml
version: 1
preset: typescript-basic
verification:
  timeoutMinutes: 20
  commands:
    - name: test
      run: pnpm test
output:
  uploadEvidenceBundle: true
  includeCommandPreview: true
  strict: true
telemetry:
  enabled: false

Current non-guarantees

Bounded means bounded.

Proofrail is explicit about what this prototype does not establish. A repository engineering review is not a product Verdict.

  • No hosted service. The public surface is a browser-readable static prototype.
  • No Marketplace listing. Installation is a copyable workflow surface, not a publication claim.
  • No trusted release claim. ADMISSIBLE applies only to the exact inputs and environment recorded in one bundle.
  • No complete sandbox. Execution is allowed only within the declared, bounded path and fails closed when enforcement is unavailable.
  • No enterprise certification. The prototype does not establish compliance, certification, or external acceptance.
  • No cloud retention. Artifacts remain with the caller-controlled workflow and local run.
  • No payment automation. Pricing is a hypothesis, not a billing system.
